# Phase 02.1: Release Hardening - Context

**Gathered:** 2026-03-21
**Status:** Ready for planning

<domain>
## Phase Boundary

Harden the already-implemented v1 product by validating core user flows, fixing release-blocking defects, and completing release-readiness checks. This phase does not introduce net-new product capabilities.

</domain>

<decisions>
## Implementation Decisions

### Hardening Scope
- Focus only on validating and fixing existing v1 functionality.
- No net-new feature capabilities are allowed in this phase.
- Work centers on reliability, correctness, and release confidence.

### Defect Prioritization
- Fix only blocker/critical issues discovered during hardening.
- Defer non-critical issues to backlog/deferred items.
- Keep fixes surgical and low-risk.

### Validation Focus
- Validate end-to-end v1 flows already delivered in phases 1-3.
- Prioritize ownership persistence, set navigation, album filtering, and stats consistency.
- Include targeted regression checks for recently modified components.

### Release Readiness Exit Criteria
- Production build must pass (`npm run build`).
- Smoke checklist must pass for core v1 paths.
- Any unresolved non-critical findings are documented as deferred follow-ups.

### Claude's Discretion
- Exact smoke checklist structure and execution order.
- Whether to add lightweight, non-invasive guardrails/tests if they reduce release risk.
- Grouping of fixes into plan(s) as long as blocker-first policy is preserved.

</decisions>

<specifics>
## Specific Ideas

- Treat this phase as “stabilize what exists,” not “expand scope.”
- Keep a strict severity threshold so release momentum is not blocked by low-impact polish requests.

</specifics>

<code_context>
## Existing Code Insights

### Reusable Assets
- Existing build/typecheck pipeline (`npm run build`) is the primary release gate.
- UI components already include loading/error/empty states and ownership interactions.
- Collection persistence hooks and set/card data adapters are already established and used.

### Established Patterns
- Hook-based async data access with explicit loading/error state handling.
- Client-side filtering/search patterns in set and album surfaces.
- localStorage-backed ownership model with derived progress/stats.

### Integration Points
- Set selection in `SetGrid` routes into `CardGrid` via `App.tsx`.
- Ownership toggles in `CardGrid` and `CardDetail` must remain consistent with set progress and stats footer.
- Hardening should verify behavior consistency across set view, album view, and persisted reload paths.

</code_context>
