# Phase 02.1: Release Hardening - Research

**Researched:** 2026-03-21
**Domain:** React application release hardening and production readiness
**Confidence:** HIGH

## Summary

Release hardening stabilizes existing v1 functionality without adding new features. This phase validates critical user flows (set browsing → card selection → ownership tracking → persistence), identifies and fixes blocker/critical defects, and ensures production readiness through build verification and smoke testing.

**Primary recommendation:** Use systematic risk hotspot analysis → defect triage by severity → targeted fixes → automated release gates (build + smoke checklist). Keep fixes surgical to minimize regression risk while maximizing release confidence.

### Anti-Patterns to Avoid
- **Scope creep during hardening:** Adding "quick features" undermines stability goal — hardening is NOT feature dev
- **Fixing cosmetic issues:** Defer polish that doesn't affect core workflow — focus keeps release on track
- **Over-testing low-risk code:** Don't write tests for trivial getters or pure UI — test risk hotspots only
- **Refactoring working code:** If it's not broken and not a blocker, don't touch it — surgical fixes only

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Component testing | Custom test harness | @testing-library/react | Handles React lifecycle, async rendering, cleanup automatically |
| User interaction simulation | Manual event dispatch | @testing-library/user-event | Real browser event sequences (focus, blur, keyboard) |
| Test runner for Vite/ESM | Jest configuration | vitest | Native Vite integration, no transform config needed |
| Manual smoke testing scripts | Bash/PowerShell checklist | Interactive checklist + vitest smoke suite | Automate what's automatable, manual for UX validation |
| localStorage mock/spy | Custom mock | vitest.spyOn or happy-dom built-in | Storage API already available in test environment |

**Key insight:** Testing infrastructure is complex — event propagation, async rendering, DOM cleanup, module mocking all have edge cases. Use battle-tested libraries that handle these correctly.

## Common Pitfalls

### Pitfall 1: localStorage Quota Exceeded Not Handled
**What goes wrong:** App crashes or silently fails when localStorage is full (5-10MB limit)
**Why it happens:** `collection.ts` persists on every change but doesn't check quota or handle errors gracefully
**How to avoid:** 
- Verify `try/catch` around `localStorage.setItem` exists (✓ already present in collection.ts:36-43)
- Check error type is `QuotaExceededError` before logging (✓ already present)
- Verify user sees error toast when quota exceeded (❌ NOT IMPLEMENTED — potential blocker)
**Warning signs:** Console errors during ownership toggle, collection changes not persisting, missing error feedback to user

### Pitfall 2: Stale Collection Data After localStorage Change
**What goes wrong:** User edits localStorage directly (devtools), app doesn't sync, shows wrong ownership
**Why it happens:** React state initialized once from localStorage, no listener for external changes
**How to avoid:**
- Add `storage` event listener to sync external changes (❌ NOT IMPLEMENTED — low priority)
- Document that app state is authoritative during session (✓ acceptable for v1)
**Warning signs:** Ownership state mismatch after localStorage manual edit, inconsistent stats

### Pitfall 3: Stats Calculation Race Conditions
**What goes wrong:** Rapid ownership toggles cause stats to desync with visible cards
**Why it happens:** Stats derived from ownedCards state, but state updates are async
**How to avoid:**
- Use `useMemo` with proper dependencies (✓ already present in CardGrid.tsx:93-99 and SetGrid.tsx:76-103)
- Verify stats computed from same state snapshot as rendered cards (✓ already correct)
**Warning signs:** Stats show 5 owned but only 4 cards have green ring, percentages flicker

### Pitfall 4: Build Succeeds Locally But Fails in Production
**What goes wrong:** `npm run build` works on dev machine, fails in CI/production, or runtime errors in production bundle
**Why it happens:** Dev mode ignores some type errors, unused imports, or circular dependencies
**How to avoid:**
- Run `npm run build` as first validation step (✓ required by exit criteria)
- Check build output for warnings (large bundle size, circular deps)
- Test production build locally (`npm run preview`) before deploying
**Warning signs:** Type errors only in build mode, "module not found" in production, blank screen after deploy

### Pitfall 5: Client-Side Filter Interactions Break UX
**What goes wrong:** Combining series filter + search + ownership filter shows wrong results or empty grid
**Why it happens:** Filters applied in wrong order, or pagination not hidden during client filtering
**How to avoid:**
- Verify filter order: API filters → series → search → ownership (SetGrid: 105-118, CardGrid: 71-90)
- Hide pagination when client filters active (✓ already implemented in SetGrid.tsx:357)
- Test ALL filter combinations in smoke checklist
**Warning signs:** Empty results when filters should match, pagination shows wrong page count

### Pitfall 6: Missing Error States in UI
**What goes wrong:** API fails (network, 404, timeout), user sees infinite loading or blank screen
**Why it happens:** Components handle loading/data states but forget error display
**How to avoid:**
- Verify error prop rendered with user-friendly message and retry button
- Check SetGrid.tsx:143-153 (✓ error state exists)
- Check CardGrid error handling (❌ VERIFY during validation)
**Warning signs:** Console errors but no UI feedback, app "stuck" on error

### Example 2: localStorage Persistence Test
```typescript
// src/lib/__tests__/collection.test.ts
import { describe, it, expect, beforeEach, afterEach } from 'vitest';
import { renderHook, act } from '@testing-library/react';
import { useCollection } from '../collection';

describe('Collection Persistence', () => {
  beforeEach(() => {
    localStorage.clear();
  });

  afterEach(() => {
    localStorage.clear();
  });

  it('persists ownership changes to localStorage', () => {
    const { result } = renderHook(() => useCollection());
    
    // Toggle ownership
    act(() => {
      result.current.addToCollection('base1-1');
    });
    
    // Verify persisted
    const stored = JSON.parse(localStorage.getItem('pokemon-collection-v2') || '{}');
    expect(stored.ownedCards['base1-1']).toBe(true);
  });

  it('loads existing collection from localStorage', () => {
    // Pre-populate localStorage
    localStorage.setItem('pokemon-collection-v2', JSON.stringify({
      version: 1,
      ownedCards: { 'base1-1': true, 'base1-2': true }
    }));
    
    // Initialize hook
    const { result } = renderHook(() => useCollection());
    
    // Verify loaded
    expect(result.current.isOwned('base1-1')).toBe(true);
    expect(result.current.isOwned('base1-2')).toBe(true);
    expect(result.current.isOwned('base1-3')).toBe(false);
  });

  it('handles localStorage quota exceeded gracefully', () => {
    const { result } = renderHook(() => useCollection());
    
    // Mock localStorage.setItem to throw QuotaExceededError
    const originalSetItem = Storage.prototype.setItem;
    Storage.prototype.setItem = () => {
      const error = new DOMException('QuotaExceededError');
      error.name = 'QuotaExceededError';
      throw error;
    };
    
    // Should not crash
    act(() => {
      result.current.addToCollection('base1-1');
    });
    
    // Restore
    Storage.prototype.setItem = originalSetItem;
  });
});
```

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| Manual QA checklists only | Automated smoke tests + manual UX validation | 2023+ | Faster feedback, regression prevention |
| jest for Vite projects | vitest native runner | 2022+ | Simpler config, faster execution |
| Detailed unit test coverage | Risk-based testing (focus hotspots) | Ongoing trend | Efficiency over completeness theater |
| Pre-deploy manual testing | Build + smoke gates in CI | Standard practice | Catch issues before production |

**Deprecated/outdated:**
- Jest for new Vite projects: requires transform config, slower than vitest
- Testing every function: modern practice focuses on integration tests and risk hotspots
- Ignoring production build validation: SSR/bundler issues only appear in build mode

## Open Questions

1. **Should we add lightweight tests for risk hotspots?**
   - What we know: No test infrastructure exists, setup cost ~30min, provides regression safety
   - What's unclear: Whether manual smoke checklist is sufficient for v1 scope
   - Recommendation: Add minimal test suite (collection persistence + stats calculation) as safety net — low cost, high value for stability

2. **How to handle TCGdex API downtime?**
   - What we know: External API, no caching layer, network failures possible
   - What's unclear: Expected uptime SLA, whether to cache responses
   - Recommendation: Verify error states render correctly, defer caching to v2 (adds complexity)

3. **Is localStorage 5MB limit realistic for large collections?**
   - What we know: Current format stores card IDs only (very small), quota error handled
   - What's unclear: How many sets user might track (100 sets × 200 cards = 20k IDs ≈ 400KB → safe)
   - Recommendation: Document limit (~50k cards before quota risk), defer export/import to v2

## Defect Triage Rubric

Use this rubric during validation to classify discovered issues:

### BLOCKER Severity (Fix Immediately)
**Definition:** Prevents core v1 value proposition — user cannot track collection

**Examples:**
- Ownership toggles don't persist after reload
- `npm run build` fails
- SetGrid shows no sets / infinite loading
- CardGrid crashes when clicking cards
- Stats footer always shows 0%

**Action:** Stop validation, fix immediately, re-validate

### CRITICAL Severity (Fix if Time Permits)
**Definition:** Severely degrades workflow but workaround exists

**Examples:**
- Stats footer shows wrong percentage (off by >10%)
- Search filter doesn't work on some sets
- API errors show no user feedback (console only)
- Completion badge shows on incomplete sets
- Ownership filter shows wrong cards

**Action:** Document clearly, fix if time permits before release, else defer to v1.1

### HIGH Severity (Defer to Backlog)
**Definition:** Noticeable degradation but doesn't block core workflow

**Examples:**
- Performance slow with 200+ card sets (>3s load)
- Minor visual bugs (alignment, spacing)
- Tooltips missing or incorrect
- Error messages unclear

**Action:** Add to backlog, document as known issue

### MEDIUM/LOW Severity (Defer to v2)
**Definition:** Polish, edge cases, or nice-to-haves

**Examples:**
- Animations could be smoother
- Better loading skeletons
- Improved empty states
- Accessibility improvements
- Mobile layout polish

**Action:** Defer to v2 scope, no action for v1
